# Schema migration contract

**Feature**: 015-source-in-timeline
**Date**: 2026-05-03

Forward-only migration per ENGINEERING.md rule 2.15. No backward-compatibility code, no migration shims, no preservation of old behavior. Schema version bumped per rule 3.1 (constitution VIII).

---

## SQL deltas — applied to `src/lua/schema.sql`

### 1. `tracks.sync_mode` — new column on existing table

```sql
ALTER TABLE tracks ADD COLUMN sync_mode TEXT
    NOT NULL
    DEFAULT 'ripple'
    CHECK (sync_mode IN ('off','ripple','cut'));
```

**Notes**:
- DEFAULT `'ripple'` matches the existing pipeline behavior, so pre-existing tracks behave identically post-migration.
- CHECK constraint enforces the enum at SQL level (rule 2.21 statically-verifiable). Bad values from buggy code path produce a SQL error before the assert in the dispatch sees them — defense in depth.

### 2. `patches` — new table

```sql
CREATE TABLE IF NOT EXISTS patches (
    id                  TEXT    PRIMARY KEY,
    sequence_id         TEXT    NOT NULL REFERENCES sequences(id) ON DELETE CASCADE,
    source_track_index  INTEGER NOT NULL CHECK (source_track_index >= 0),
    record_track_index  INTEGER NOT NULL CHECK (record_track_index >= 0),
    enabled             INTEGER NOT NULL DEFAULT 1 CHECK (enabled IN (0,1)),
    UNIQUE (sequence_id, source_track_index)
);

CREATE INDEX IF NOT EXISTS idx_patches_sequence_id
    ON patches(sequence_id);
```

**Notes**:
- `id` is a TEXT UUID generated by the command executor (matches JVE's existing convention).
- CASCADE on sequence delete keeps `patches` in lockstep with their owning sequence; no orphan rows possible.
- No FK on `record_track_index` — the index MAY exceed the active sequence's current track count; auto-create at edit time (FR-029b) handles that.
- No `color_index` column (per spec FR-031 — patches have no per-patch color; indicators are role-colored).
- Index on `sequence_id` for the per-sequence query in patch-resolution (every edit reads "all patches for this sequence").

### 3. Schema version bump

```sql
INSERT INTO schema_version (version, applied_at)
    VALUES (<NEW_VERSION>, strftime('%s','now'));
```

**Notes**:
- `<NEW_VERSION>` is the next integer after the current schema_version row. /tasks pins the actual number.
- No migration code preserves prior behavior (rule 2.15). Old projects on the prior schema version are NOT auto-migrated by JVE — the user resets the project DB per the existing JVE workflow (CLAUDE.md "Project Database" section).

### 4. NO change to `snapshots` table

Per Clarifications 2026-05-03 Q1 + plan research R8: the 6 non-undoable toggle commands write directly to their target tables and produce NO snapshot rows. The `snapshots.clips_state` JSON shape is unchanged.

### 5. NO change to `clip_links` table

`clip_links` (existing V+A clip-level linkage) is independent of new `patches`. No schema change. FR-021b enforces the conceptual separation; no schema-level enforcement needed because they share no columns.

### 6. NO change to `commands` table

The `commands` table schema is unchanged. The `undoable = false` SPEC flag (R1) influences the dispatcher's behavior, not the table schema. The flag MAY be reflected via leaving `undo_group_id` NULL — that's an implementation detail handled in /tasks.

## Migration test contract

A test `tests/test_schema_migration_015.lua` MUST verify, in this order:

1. Pre-migration sanity: open a project on the prior schema version.
2. Apply the migration (`schema.sql` re-execution against the existing DB OR a dedicated migration script — implementation choice in /tasks).
3. Post-migration:
   - `tracks.sync_mode` column exists with CHECK constraint enforced (insert with bad value → SQL error).
   - `patches` table exists with documented columns and constraints.
   - `UNIQUE(sequence_id, source_track_index)` enforced (insert duplicate → SQL error).
   - CASCADE on sequence delete: insert a sequence + patch, delete the sequence, verify the patch row is gone.
   - `record_track_index` may exceed the sequence's track count (insert succeeds — no FK).
   - `schema_version` table reports the new version.
   - `snapshots` table is unchanged (no new column).
   - `clip_links` table is unchanged.
4. Existing tracks have `sync_mode = 'ripple'` (default applied by ALTER).

This migration is independent of any other feature's pending migrations. Within /tasks, the migration runs as task #1 (schema first). All subsequent tasks assume the post-migration schema.
