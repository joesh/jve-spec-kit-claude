# Phase 1 Data Model: Source-in-Timeline + Track-Header Redesign + Tristate Sync-Lock

**Feature**: 015-source-in-timeline
**Date**: 2026-05-03

This document is the canonical specification of all entity / schema / state-shape changes introduced by this feature. The schema delta itself lives in [`contracts/schema-migration.md`](./contracts/schema-migration.md); this document explains the intent and validation rules.

---

## 1. New entities

### 1.1 `Patch`

A per-record-sequence routing binding from one source-track-index to one record-track-index, with an on/off enabled flag.

| Field | Type | Constraint | Notes |
|---|---|---|---|
| `id` | TEXT | PRIMARY KEY | UUID-style id, generated by command executor. |
| `sequence_id` | TEXT | NOT NULL, FK → `sequences(id)` ON DELETE CASCADE | The record sequence this patch belongs to. |
| `source_track_index` | INTEGER | NOT NULL, ≥ 0 | Index into the source's master sequence track list (0 = video V1, 1 = audio A1, etc., matching JVE's existing track indexing). |
| `record_track_index` | INTEGER | NOT NULL, ≥ 0 | Destination index on `sequence_id`'s track list. Defaults to `source_track_index` (identity routing) on creation. |
| `enabled` | INTEGER | NOT NULL, DEFAULT 1, CHECK IN (0,1) | The on/off button state. ON = include in edits; OFF = drop. |

**Identity & uniqueness**: `UNIQUE(sequence_id, source_track_index)` — one patch per (record-sequence, source-track-index) pair. There is at most one patch governing where source-track N goes when this sequence is the active record.

**Lifecycle**:
- Created lazily on first user interaction with a source track (toggle, drag) for which no patch row exists. Default: `enabled=1`, `record_track_index=source_track_index`.
- Updated by `SetPatch` command (FR-035a) — non-undoable per FR-040.
- Deleted on parent sequence deletion (CASCADE).
- Patches do NOT automatically prune when source has fewer tracks than expected — they remain dormant. A patch with `source_track_index=7` is harmless when the loaded source has only 4 tracks; it will activate if a 5+-track source is loaded later.

**Validation rules**:
- `record_track_index` MAY exceed the active sequence's current track count. At edit time (FR-029b) the auto-create-record-track step extends the sequence as needed. The patch row itself does NOT need to gate on track existence.
- Cross-track-type drag (audio source → video record, or vice versa) MUST be refused at the command layer (FR-010a, rule 2.13). The schema does not enforce track-type matching because track-type lookup requires joining to source's master sequence; that is the command executor's responsibility.

**Relationships**:
- `patches` is INDEPENDENT of `clip_links`. `clip_links` (existing) is V+A clip-level linkage; `patches` is track-level routing. They share no schema.

### 1.2 `Track.sync_mode` (column on existing `tracks` table)

| Field | Type | Constraint | Notes |
|---|---|---|---|
| `sync_mode` | TEXT | NOT NULL, DEFAULT `'ripple'`, CHECK IN (`'off'`, `'ripple'`, `'cut'`) | Per-track ripple-behavior mode. |

**Lifecycle**:
- New tracks default to `'ripple'` (matches existing pipeline behavior — FR-023).
- Toggled by `SetSyncMode` command (FR-024) — non-undoable per FR-040.
- Persists with the track row; restored on reopen (FR-028).

**Validation rules**:
- `CHECK` constraint enforces enum membership at SQL level (rule 2.21 statically-verifiable).
- The ripple-pipeline dispatch (FR-026) reads `track.sync_mode` and dispatches; an unrecognized value MUST `assert()` immediately with the track id and the bad value (FR-047.2).

## 2. Existing entities consulted (no schema change)

### 2.1 `tracks.muted`, `tracks.soloed`, `tracks.locked`, `tracks.enabled`

All four already exist (`schema.sql:160-163`). No schema change. Behavior change ONLY: their toggle commands become non-undoable per FR-040 (was: undoable, FR-040a bug).

### 2.2 `tracks.volume`, `tracks.pan`

Existing columns (`schema.sql:165-166`). Behavior unchanged. Out of scope for this feature.

### 2.3 `sequences` (`master` vs `nested`)

Existing kind discriminator (`schema.sql:84-85`). No change. The Source tab represents whichever master sequence is loaded in the source monitor — relies on existing `kind = 'master'` semantics from feature 013.

### 2.4 `clip_links`

Existing V+A clip-level linkage (`schema.sql:281`). Out of scope. NOT to be conflated with new `patches` (FR-021b).

### 2.5 `snapshots`

Existing schema unchanged (R8 finding). No `clips_state` extension; no new columns. The 6 non-undoable toggle commands write directly to their target tables and produce no `snapshots` row.

### 2.6 `commands` (table)

Existing schema unchanged. The 6 non-undoable toggle commands MAY still write a `commands` row (so the dispatcher logs them) but the command_manager extension (R1) ensures they are not pushed to the per-sequence undo stack — implementation detail deferred to /tasks (could be: write the row but exclude it from the undo cursor; or skip the row entirely).

## 3. UI-only state pointers (not persisted in DB; managed in `timeline_state`)

### 3.1 `displayed_tab_id`

The tab whose content the timeline body is currently rendering. Either a `sequence_id` (Record tab or Source tab — both keyed by sequence_id since SourceTab IS a tab whose sequence is the source-monitor's loaded master) or NULL (no tab open).

### 3.2 `active_sequence_id`

The sequence whose tab was last selected. Edit operations target this. NEVER set to the SourceTab's master sequence_id (FR-005). NULL only at app-start when no sequence is open.

**Persistence**: `active_sequence_id` persists across project sessions via the existing `last_open_sequence_id` mechanism in `project_settings`. `displayed_tab_id` MAY persist similarly (extend `project_settings`) so reopening a project restores the same view.

|---|---|---|---|
| `source_routing_view` | TEXT enum | `'per_channel'` | Values: `'per_channel'` (default — N source buttons, FR-029c-A) or `'per_clip'` (one button representing the whole source audio, FR-029c-B). |

## 5. State transitions

### 5.1 Patch lifecycle

```
[no row]  --user toggles ON or drags-->  [enabled=1, record_track_index=identity-default]
[any]     --user toggles OFF-->          [enabled=0]                                    (record_track_index unchanged)
[any]     --user toggles ON-->           [enabled=1]                                    (record_track_index unchanged)
[any]     --user plain-drags onto rec X-->   [enabled=1, record_track_index=X]
[any]     --user modifier-drags onto rec X-->   creates an additional patch row (UNIQUE constraint on (sequence_id, source_track_index) prevents this directly — see clarification below)
```

**Clarification on stacking**: FR-010a's "modifier-drag = stack" semantics combined with `UNIQUE(sequence_id, source_track_index)` means stacking is NOT N rows on the patches table. The patch table holds at most one row per (sequence, source_track_index). To support "multiple sources targeting one record track" — i.e., multi-source-to-one-track — the existing schema already permits two patches with different `source_track_index` and the same `record_track_index`:

```
patches:
  (seq=S, source_track_index=1, record_track_index=1, enabled=1)  -- src A1 → rec A1
  (seq=S, source_track_index=2, record_track_index=1, enabled=1)  -- src A2 → rec A1 (stacked)
  (seq=S, source_track_index=3, record_track_index=2, enabled=1)  -- src A3 → rec A2
```

Stacking via FR-010a modifier-drag updates the dragged patch's `record_track_index` to match the destination's existing `record_track_index`. The result: the dragged source now also targets that record track, joining whatever was already there. Multi-channel clip production at edit time is the responsibility of the edit command's executor, which queries `patches` for all rows where `record_track_index = N` to determine the channel composition.

### 5.2 Sync-mode lifecycle

```
[track created with sync_mode='ripple']  --click sync cell-->  [sync_mode='cut']
[sync_mode='cut']                         --click sync cell-->  [sync_mode='off']
[sync_mode='off']                         --click sync cell-->  [sync_mode='ripple']
```

Cycle: Off → Ripple → Cut → Off (FR-024). Each transition is a non-undoable command write.

### 5.3 SourceTab visibility

```
[no source loaded, tab closed]  --source_viewer.load_master_clip(M)-->  [tab not auto-opened; source loaded; previous closed-state preserved]
[no source loaded, tab open]    --source_viewer.load_master_clip(M)-->  [tab now displays M; was empty placeholder]
[source loaded, tab closed]     --user invokes "Show Source Tab"-->     [tab opens; displays current loaded source]
[source loaded, tab open]       --user clicks ×-->                       [tab closes; source remains loaded]
[source loaded, tab open]       --source unloaded-->                     [tab persists; renders empty placeholder per FR-007b]
```

## 6. Validation matrix (assert sites — supports FR-047)

| Site | Assert | Failure message must include |
|---|---|---|
| Patch lookup | row exists for given `(sequence_id, source_track_index)` if expected | sequence_id, source_track_index |
| Sync_mode dispatch | `track.sync_mode IN ('off','ripple','cut')` | track_id, observed sync_mode value |
| SourceTab display | source_monitor has loaded master before render | "no source loaded" + tab id |
| Displayed-tab switch | target tab's sequence_id exists in DB | sequence_id |
| Patch command | command's sequence_id is set; `source_track_index` and `record_track_index` are in valid ranges | command name, all three values |
| Edit-time routing | each source patch's `enabled` was consulted before include/exclude | source_track_index per patch read |
| 3-point math | at least 3 of 4 marks set before computing 4th | which marks are present |
| Patch/sync-mode command execution | command is `non_undoable`-flagged AND writes directly to its target table | command name, target table |

All asserts include the function/module name (`SetPatch`, `SetSyncMode`, `apply_per_track_sync_mode_dispatch`, etc.) so the crash site is obvious (rule 1.14).

## 7. Test surface (data-level)

Per FR-048, these data-level tests are mandatory:

1. Schema migration: post-migration, `tracks` has `sync_mode` column with default `'ripple'`; `patches` table exists with the documented columns.
2. `Patch` UNIQUE constraint: inserting two rows with same `(sequence_id, source_track_index)` produces a SQL error.
3. `Patch.record_track_index` may exceed sequence track count (no FK to a track-index range).
4. `tracks.sync_mode` CHECK rejects values outside `('off','ripple','cut')`.
5. CASCADE on sequence delete: deleting a sequence removes its `patches` rows.
6. `snapshots` table is unchanged by any of the 6 non-undoable toggle commands.
7. Schema version bump: post-migration, the `schema_version` table reports the new version.

These appear as concrete test cases in `tasks.md` (Phase 2).
